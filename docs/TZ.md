# Техническое Задание (ТЗ)

## Платформа «ШторБаза» — внутренняя система управления ассортиментом штор и фурнитуры

---

## 1. Общее описание проекта

### 1.1. Название

**ШторБаза** — внутренняя веб-платформа для компании по продаже штор и сопутствующих товаров.

### 1.2. Краткое описание

Веб-платформа для быстрого поиска товаров, актуальных цен и остатков. Заменяет звонки менеджерам и устраняет путаницу с единицами измерения (метры/штуки) и ценами.

### 1.3. Проблема

| Проблема | Последствия |
|----------|-------------|
| Сотрудники звонят менеджерам, чтобы узнать цену и наличие | Менеджеры перегружены, высокий риск ошибок |
| Товары имеют разные единицы измерения (метр, штука, упаковка) | Путаница в ценах, неправильные расчёты для клиентов |
| Ассортимент большой, информация разрозненна | Долгий поиск, потеря времени, ошибки в заказах |
| Нет истории изменения цен и остатков | Невозможно установить, кто допустил ошибку |

### 1.4. Цель

Создать шикарную, понятную и безопасную платформу, где:

- Сотрудник **за 5 секунд** находит товар с ценой и остатком
- **Невозможно перепутать** единицу измерения — она визуально выделена
- Менеджер управляет ценами и остатками с **полным логированием**
- Администратор контролирует всю систему

### 1.5. Ключевые метрики успеха (KPI)

| Метрика | Целевое значение |
|---------|-----------------|
| Время поиска товара сотрудником | ≤ 5 секунд |
| Количество звонков менеджерам по цене/наличию | Снижение на 90% |
| Ошибки в ценах при оформлении заказов | Снижение на 95% |
| Время обновления цены менеджером | ≤ 30 секунд |
| Время загрузки страницы каталога | ≤ 1.5 секунды |

## 3. Функциональные требования

### 3.1. Модуль «Аутентификация и авторизация»

**User Stories:**
- Как сотрудник, я хочу войти по логину и паролю, чтобы видеть каталог
- Как администратор, я хочу приглашать новых пользователей по email, чтобы контролировать доступ

**Описание:**
- Вход по email + пароль через Supabase Auth
- Регистрация **только по приглашению** администратора (закрытая система)
- При входе определяется роль пользователя из таблицы `profiles`
- Сессия хранится в httpOnly cookie (управляется Supabase SSR)
- При неактивности 8 часов — автоматический выход

**Бизнес-правила:**
- Самостоятельная регистрация запрещена
- Пароль: минимум 8 символов, буквы + цифры
- После 5 неудачных попыток входа — блокировка на 15 минут
- Администратор может деактивировать учётную запись (не удалять)

### 3.2. Модуль «Каталог товаров»

**User Stories:**
- Как сотрудник, я хочу видеть каталог товаров с ценами и остатками, чтобы быстро ответить клиенту
- Как сотрудник, я хочу сразу понять — цена за метр или за штуку, чтобы не ошибиться
- Как менеджер, я хочу добавить новый товар с указанием единицы измерения

**Описание:**

Каталог отображается в виде таблицы (основной режим) или сетки карточек (опционально). Каждый товар содержит:

| Поле | Описание | Обязательное |
|------|----------|-------------|
| Название | Наименование товара | ✅ |
| Артикул | Уникальный код товара | ✅ |
| Категория | Шторы, Карнизы, Фурнитура, Ткани, Ленты, Аксессуары | ✅ |
| Цена | Числовое значение в тенге (₸) | ✅ |
| Единица измерения | `метр` или `штука` | ✅ |
| Остаток | Количество в соответствующих единицах | ✅ |
| Заметка «как считать» | Текстовая подсказка для сотрудника | ❌ |
| Изображение | Фото товара (до 3 шт.) | ❌ |
| Описание | Текстовое описание | ❌ |
| НДС | Включён в цену или нет | ✅ |
| Статус | Активный / Скрытый / Снят с продажи | ✅ |

### 3.4. Модуль «Поиск и фильтрация»

**User Stories:**
- Как сотрудник, я хочу найти товар за 5 секунд по названию или артикулу
- Как сотрудник, я хочу фильтровать по категории и единице измерения

**Описание:**

| Тип поиска | Описание |
|------------|----------|
| Полнотекстовый поиск | По названию, артикулу, описанию. С подсветкой найденного текста |
| Фильтр по категории | Выпадающий список с множественным выбором |
| Фильтр по единице измерения | «Все» / «Метровые» / «Штучные» |
| Фильтр по статусу | «Активные» / «Все» / «Скрытые» / «Снятые с продажи» |
| Фильтр по наличию | «В наличии» / «Заканчивается» (< 10) / «Нет в наличии» |
| Сортировка | По названию, цене, остатку, дате обновления |
| Диапазон цен | Ползунок «от — до» |

**Бизнес-правила:**
- Поиск начинается с 2-го символа (debounce 300мс)
- Результаты обновляются без перезагрузки страницы
- Параметры фильтров сохраняются в URL (можно поделиться ссылкой)
- Фильтры комбинируются по логическому «И»

### 3.5. Модуль «Управление товарами» (Менеджер, Администратор)

**User Stories:**
- Как менеджер, я хочу обновить цену товара и быть уверен, что изменение зафиксировано
- Как менеджер, я хочу быстро обновить остатки после поставки

**Описание:**

**Форма редактирования товара:**

| Поле | Тип | Валидация |
|------|-----|-----------|
| Название | Текст | 2–200 символов, обязательное |
| Артикул | Текст | Уникальный, 2–50 символов, только латиница + цифры + дефис |
| Категория | Выбор из списка | Обязательное |
| Цена | Число | > 0, до 2 знаков после запятой |
| Единица измерения | Выбор: метр / штука | Обязательное, **нельзя менять после создания** (защита от путаницы) |
| Остаток | Число | ≥ 0, для метровых — до 1 знака после запятой |
| НДС | Переключатель | Включён / Не включён |
| Заметка | Текст | До 500 символов |
| Описание | Текст | До 2000 символов |
| Изображения | Загрузка файлов | До 3 шт., макс 5 МБ каждое, JPG/PNG/WebP |
| Статус | Выбор | Активный / Скрытый / Снят с продажи |

**Бизнес-правила:**
- Единица измерения **фиксируется при создании товара** и не может быть изменена. Для изменения — создаётся новый товар
- При сохранении изменений создаётся запись в логе
- Менеджер может редактировать только: цену, остаток, заметку, описание, изображения, статус
- Удаление товара — только Администратор, с подтверждением
- «Мягкое» удаление: товар скрывается, не удаляется из БД

**Массовое обновление остатков:**
- Загрузка Excel-файла с колонками: Артикул, Новый остаток
- Предпросмотр изменений перед применением
- Каждое изменение логируется

### 3.6. Модуль «Логирование изменений» (Аудит)

**User Stories:**
- Как администратор, я хочу видеть, кто и когда изменил цену товара, чтобы разобрать спорные ситуации
- Как менеджер, я хочу видеть историю своих изменений, чтобы проверить себя

**Описание:**

Каждое изменение товара записывается в таблицу `audit_log`:

| Поле | Описание | Пример |
|------|----------|--------|
| Дата и время | Когда произошло изменение | 2026-04-13 14:32:05 |
| Пользователь | Кто изменил | Алия Кенесова (менеджер) |
| Товар | Артикул + название | SH-0421 — Штора «Венеция» |
| Поле | Какое поле изменено | Цена |
| Старое значение | Что было | 4 000 ₸ |
| Новое значение | Что стало | 4 500 ₸ |
| Комментарий | Причина (опционально) | Поставщик поднял цену |

**Бизнес-правила:**
- Логи нельзя редактировать или удалять (даже администратору)
- Хранение логов — минимум 2 года
- Менеджер видит только свои логи
- Администратор видит все логи
- Фильтрация логов: по дате, пользователю, товару, типу изменения

## 4. Нефункциональные требования

### 4.1. Производительность

| Параметр | Требование |
|----------|-----------|
| Загрузка каталога (до 5000 товаров) | ≤ 1.5 секунды |
| Поиск | ≤ 300 мс после debounce |
| Сохранение изменений | ≤ 1 секунда |
| Экспорт Excel (5000 товаров) | ≤ 5 секунд |
| Одновременных пользователей | До 50 |

### 4.2. Безопасность

| Мера | Описание |
|------|----------|
| Аутентификация | Supabase Auth, email + пароль, httpOnly cookies |
| Авторизация | Row Level Security (RLS) в Supabase на уровне БД |
| CSRF-защита | Встроенная через Supabase SSR |
| XSS-защита | Санитизация ввода, экранирование вывода (React по умолчанию) |
| SQL-инъекции | Параметризованные запросы через Supabase SDK |
| Rate Limiting | Ограничение запросов на уровне Supabase и middleware Next.js |
| Шифрование | HTTPS/TLS для всех соединений |
| Хранение паролей | bcrypt через Supabase Auth (автоматически) |
| Загрузка файлов | Проверка MIME-типа, ограничение размера, Supabase Storage с RLS |
| Инвайт-система | Регистрация только по приглашению, невозможна самостоятельная |
| Логи неизменяемы | INSERT-only таблица, без UPDATE/DELETE прав |

### 4.3. Доступность и совместимость

| Параметр | Требование |
|----------|-----------|
| Браузеры | Chrome, Firefox, Edge, Safari (последние 2 версии) |
| Устройства | Десктоп (основной), планшет, мобильный |
| Подход | Desktop-first, адаптивный дизайн через Tailwind CSS |
| Доступность | Семантическая разметка, навигация клавиатурой, ARIA-атрибуты |

### 4.4. Отказоустойчивость

| Параметр | Описание |
|----------|----------|
| Бэкап БД | Автоматический через Supabase (ежедневно) |
| Downtime | Supabase: 99.9% SLA (на Pro плане) |
| Обработка ошибок | Graceful degradation, информативные сообщения пользователю |

### 5.3. Принципы архитектуры

- **Server Components по умолчанию** — данные загружаются на сервере
- **Server Actions для мутаций** — создание/обновление товаров через серверные функции
- **Client Components только для интерактивности** — поиск, фильтры, формы
- **RLS на уровне БД** — даже при ошибке в коде, данные защищены
- **Валидация на двух уровнях** — Zod на клиенте + CHECK constraints в БД

### 6.4. RLS-политики (Row Level Security)

```sql
-- Товары: все авторизованные пользователи могут читать активные
CREATE POLICY "Чтение активных товаров" ON products
  FOR SELECT USING (
    status = 'active' OR
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role IN ('manager', 'admin'))
  );

-- Товары: менеджеры и админы могут создавать и обновлять
CREATE POLICY "Менеджеры создают товары" ON products
  FOR INSERT WITH CHECK (
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role IN ('manager', 'admin'))
  );

CREATE POLICY "Менеджеры обновляют товары" ON products
  FOR UPDATE USING (
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role IN ('manager', 'admin'))
  );

-- Логи: INSERT-only, нет UPDATE/DELETE
CREATE POLICY "Запись логов" ON audit_log
  FOR INSERT WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Менеджер видит свои логи" ON audit_log
  FOR SELECT USING (
    user_id = auth.uid() OR
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
  );

-- Профили: каждый видит свой, админ — все
CREATE POLICY "Чтение профилей" ON profiles
  FOR SELECT USING (
    id = auth.uid() OR
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
  );
```

## 7. API — Server Actions

Вместо REST API используются **Next.js Server Actions** (серверные функции) для всех мутаций. Чтение данных происходит через Server Components напрямую из Supabase.

### 7.1. Действия с товарами

| Действие | Функция | Входные данные | Роли |
|----------|---------|----------------|------|

| Удалить товар (мягкое) | `deleteProduct(id)` | UUID | Админ |

## 9. Логирование и аудит

### 9.1. Что логируется

| Событие | Данные |
|---------|--------|
| Создание товара | Все поля нового товара |
| Изменение цены | Старая цена → новая цена |
| Изменение остатка | Старый остаток → новый остаток |
| Изменение статуса | Старый статус → новый статус |
| Изменение заметки | Старый текст → новый текст |
| Удаление товара | Артикул + название (мягкое удаление) |
| Массовое обновление | Каждая строка — отдельная запись |
| Вход пользователя | email, дата, IP (через Supabase Auth Logs) |

### 9.2. Реализация

Логирование реализуется через **PostgreSQL триггерную функцию** на таблице `products`:

```sql
CREATE OR REPLACE FUNCTION log_product_changes()
RETURNS TRIGGER AS $$
BEGIN
  -- При UPDATE: для каждого изменённого поля создаётся запись
  IF TG_OP = 'UPDATE' THEN
    IF OLD.price IS DISTINCT FROM NEW.price THEN
      INSERT INTO audit_log (user_id, product_id, field_name, old_value, new_value, action)
      VALUES (auth.uid(), NEW.id, 'price', OLD.price::text, NEW.price::text, 'update');
    END IF;
    -- Аналогично для stock, note, status, name, description
  END IF;

  IF TG_OP = 'INSERT' THEN
    INSERT INTO audit_log (user_id, product_id, field_name, old_value, new_value, action)
    VALUES (auth.uid(), NEW.id, 'product', NULL, NEW.name, 'create');
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER trg_product_audit
  AFTER INSERT OR UPDATE ON products
  FOR EACH ROW EXECUTE FUNCTION log_product_changes();
```

### 9.3. Защита логов

```sql
-- Никто не может обновлять или удалять логи
REVOKE UPDATE, DELETE ON audit_log FROM authenticated;
REVOKE UPDATE, DELETE ON audit_log FROM service_role;
```

## 13. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Артикул (SKU)** | Уникальный буквенно-цифровой код товара (например, SH-0421) |
| **Единица измерения** | Метр или штука — определяет, в чём измеряется цена и остаток |
| **Метровый товар** | Товар, цена которого указана за 1 погонный метр (ткани, шторы, ленты) |
| **Штучный товар** | Товар, цена которого указана за 1 единицу (крючки, люверсы, карнизы) |
| **НДС** | Налог на добавленную стоимость (12% в РК) |
| **RLS** | Row Level Security — безопасность на уровне строк БД в PostgreSQL |
| **Server Action** | Серверная функция Next.js для мутации данных |
| **Server Component** | Компонент React, выполняющийся на сервере (для чтения данных) |
| **Мягкое удаление** | Товар не удаляется из БД, а помечается статусом «Снят с продажи» |
| **Аудит-лог** | Неизменяемая запись о каждом действии пользователя в системе |
| **Инвайт** | Приглашение нового пользователя в систему по email |
| **Прайс-лист** | Таблица с ценами и описанием товаров для клиентов |
| **Tooltip** | Всплывающая подсказка при наведении курсора на элемент |
